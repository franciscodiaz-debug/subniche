# v0 Import Workflow

> **Purpose:** Step-by-step protocol for importing code generated by v0.dev into this project.
> Read this document in full before touching any v0-generated code.

---

## What is v0?

[v0.dev](https://v0.dev) is Vercel's AI design tool that generates React/Next.js components using shadcn/ui and Tailwind CSS. The output is almost — but not exactly — compatible with this project. This workflow closes the gap.

---

## Step 0 — Understand what was pasted

Before anything else, scan the v0 code and answer:

1. **What does it render?** (screen, modal, card, form, etc.)
2. **Is it interactive?** (has `onClick`, `onChange`, `useState`, etc.)
3. **What shadcn components does it import?** (Button, Card, Input, etc.)
4. **What icons does it use?** (should be `lucide-react`)
5. **Does it fetch data or just display it?**
6. **Does it use `cn()` from `@/lib/utils`?**

Do not skip this. The audit determines everything that follows.

---

## Step 1 — Install any missing shadcn components

v0 often uses shadcn components not yet in this project. Check against `components/ui/`:

```bash
ls components/ui/
```

For any component imported in v0 code but missing from `components/ui/`, install it:

```bash
npx shadcn@latest add <component-name>
```

Examples: `badge`, `select`, `dialog`, `popover`, `tooltip`, `tabs`, `avatar`, `separator`

> ⚠️ Never copy shadcn component source from v0 manually. Always use the CLI.

---

## Step 2 — Fix Tailwind CSS 4 compatibility

This project uses **Tailwind CSS 4**. v0 sometimes generates Tailwind CSS 3 syntax. Check for these and fix:

| v0 output (Tailwind 3) | Replace with (Tailwind 4) |
|------------------------|---------------------------|
| `border-border` | `border` (auto-inherits from `--border`) |
| `bg-background` | `bg-background` ✅ same |
| `text-foreground` | `text-foreground` ✅ same |
| `ring-offset-background` | `outline-offset-*` or remove |
| `focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2` | `focus-visible:ring-2 focus-visible:ring-ring` |
| arbitrary `[&_*]` selectors | Usually safe, keep as-is |

Key rule: if a class references a CSS variable token (like `bg-primary`, `text-muted-foreground`) it will work — those are defined in `app/globals.css`.

---

## Step 3 — Map CSS custom properties

v0 uses standard shadcn variable names. This project extends them with a custom `--brand` variable (green/yellow accent). Map when appropriate:

| Variable | Meaning in this project |
|----------|------------------------|
| `--background` | Page/white background |
| `--foreground` | Dark text |
| `--primary` | Primary action color |
| `--brand` | Brand accent — use for highlights, CTAs |
| `--muted` | Disabled / inactive UI |
| `--destructive` | Error / delete actions |
| `--border` | Borders |
| `--ring` | Focus ring |

If v0 uses `accent` or `popover` classes that seem off visually, consider replacing with `--brand`.

---

## Step 4 — Fix the className pattern

v0 uses `cn()` from `@/lib/utils` — this project has it too, so `cn()` calls are fine.

**But** feature components (non-UI primitives) should prefer the `array.join(" ")` pattern from `frontend.md`:

```tsx
// ✅ For feature components (components/{feature}/)
className={[
  "base-classes",
  condition ? "active-class" : "inactive-class",
  className,
].join(" ")}

// ✅ For UI primitives (components/ui/) — cn() is fine here
className={cn("base-classes", variantClass, className)}
```

If the v0 component belongs in `components/ui/` (generic primitive), keep `cn()`.  
If it belongs in `components/{feature}/` (domain-specific), convert to `array.join()`.

---

## Step 5 — Decide the component type and location

| If the component is... | Place it in... |
|------------------------|----------------|
| A generic primitive (button-like, card-like, badge, etc.) | `components/ui/` — only via `npx shadcn@latest add` |
| Domain-specific (item card, listing row, user profile block) | `components/{feature}/` |
| A full page layout | `app/{route}/page.tsx` or a dedicated layout component |

Never paste v0 output directly into `components/ui/` — those are managed by the shadcn CLI.

---

## Step 6 — Fix the `"use client"` directive

v0 often adds `"use client"` to everything. Only keep it if the component actually needs it:

**Keep `"use client"` if:**
- Uses `useState`, `useEffect`, `useCallback`, `useRef`, `useReducer`
- Has `onClick`, `onChange`, `onSubmit` handlers
- Uses browser APIs (`window`, `document`, `localStorage`)
- Uses `useRouter`, `useSearchParams`, `usePathname`

**Remove `"use client"` if:**
- Component just renders static markup
- Data flows in via props from a server component
- No interactivity or hooks

---

## Step 7 — Apply TypeScript standards

v0 usually generates decent TypeScript, but verify:

```tsx
// ✅ Props must extend HTML element types
interface CardProps extends React.ComponentProps<typeof BaseCard> {
  variant?: "default" | "featured"
}

// ✅ Named exports only
export function ItemCard({ variant = "default", className, ...props }: CardProps) { }

// ❌ No default exports
export default function ItemCard() { } // <- fix this

// ❌ No 'any'
props: any // <- fix this
```

---

## Step 8 — Apply accessibility fixes

v0 is decent but not perfect on a11y. Check:

- Images: always have `alt` text
- Icon-only buttons: need `aria-label`
- Form inputs: need `<label htmlFor>` or `aria-label`
- Error messages: need `role="alert"`
- Decorative icons: need `aria-hidden="true"`

---

## Step 9 — Final validation checklist

Before calling the task done, verify:

- [ ] Missing shadcn components were installed via CLI
- [ ] Tailwind CSS 4 syntax is correct (no ring-offset issues)
- [ ] `cn()` vs `array.join()` used per component type
- [ ] `"use client"` is present only where needed
- [ ] File is named in `kebab-case.tsx`
- [ ] Exports are named (not default)
- [ ] Props are typed with TypeScript interfaces
- [ ] No `any` types
- [ ] Accessibility attributes included
- [ ] Placed in the correct directory (`components/{feature}/`)
- [ ] ESLint passes: `npm run lint`

## Common v0 patterns and their fixes

### Pattern: data hardcoded in component

v0 often hardcodes mock data inside the component. Extract it to props:

```tsx
// v0 generated
export function ItemList() {
  const items = [{ id: 1, name: "Item 1" }, { id: 2, name: "Item 2" }]
  return <ul>{items.map(i => <li key={i.id}>{i.name}</li>)}</ul>
}

// ✅ Fixed — data as props
interface ItemListProps {
  items: Array<{ id: number; name: string }>
}
export function ItemList({ items }: ItemListProps) {
  return <ul>{items.map(i => <li key={i.id}>{i.name}</li>)}</ul>
}
```

### Pattern: inline event handlers with fetch

v0 sometimes generates fetch calls inside components. Move to server actions or keep in a handler function:

```tsx
// ✅ Keep the fetch in a named handler
const handleSubmit = async (event: React.FormEvent<HTMLFormElement>) => {
  event.preventDefault()
  // ... fetch logic following async-operations pattern from frontend.md
}
```

### Pattern: v0 uses `import { cn } from "@/lib/utils"` but file doesn't exist yet

Check if `lib/utils.ts` exists. In this project it does ✅. If it ever doesn't:

```bash
# Already exists in this project at lib/utils.ts
```

### Pattern: v0 imports a non-existent hook

v0 sometimes generates `useToast`, `useMobile`, etc. that don't exist in the project. Check if the hook exists:

```bash
ls components/ui/ | grep hook
```

If missing, either install via shadcn (`npx shadcn@latest add use-toast`) or remove and replace with the existing pattern.

## Quick-start example

User pastes v0 code for a "Item Card" component. Here's the flow:

1. Scan: it's a card with a button — interactive, needs `"use client"` ✅
2. Imports `Badge` and `Card` — check `ls components/ui/` → `badge` is missing → `npx shadcn@latest add badge`
3. Uses `cn()` — keep it, it's in `lib/utils.ts`
4. Has hardcoded data → extract to props
5. Named export ✅, TypeScript types ✅
6. Place in `components/items/item-card.tsx`
7. Run `npm run lint` — passes ✅
